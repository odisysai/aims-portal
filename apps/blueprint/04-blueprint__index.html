AG-UI Transport Layer

Persistent agent session, event streaming, and message lifecycle

AG-UI provides the persistent session envelope. Every agent interaction is a thread — with threadId, ordered message snapshots, run events, and state checkpoints. The UI never owns truth; the thread store does.

My Addition — Thread Persistence Strategy

Replace sessionStorage with a durable thread store. Recommended: Redis for hot threads (last 24h active), PostgreSQL for cold thread archive. Thread hydration on reconnect must replay the last N state snapshots to restore workspace context. Inspector/Gemma threads are persisted locally (SQLite) and merged on sync via vector clock reconciliation.

// AG-UI Thread Envelope
interface AIMSThread {
  threadId: string;          // UUIDv7 — time-ordered
  role: 'planner' | 'reviewer' | 'supervisor' | 'inspector';
  agentId: string;           // Which agent owns this thread
  deviceId?: string;         // Set for Inspector/Gemma threads
  messages: MessageSnapshot[];
  state: WorkspaceState;
  runs: RunRecord[];
  createdAt: string;         // ISO 8601
  lastActiveAt: string;
  persistenceLayer: 'redis' | 'postgres' | 'sqlite-local';
  syncVector?: CRDTVector;   // Inspector only
}

A2UI Dynamic Workspace Layer

Agent-proposed UI · Deterministic renderer · Authorization gate

The agent may propose the UI and action. Authorization and execution remain in deterministic backend services. A2UI is the bridge — it receives a COMPONENT_RENDER event, checks the Authorization Matrix, resolves the component from the trusted catalog, hydrates props, and mounts the component. The agent cannot render arbitrary HTML.

A2UI Render Pipeline

1
Agent emits intent → resolves to componentId
2
COMPONENT_RENDER event emitted via AG-UI SSE
3
Authorization Gate: role × componentId × action → ALLOW / DENY
4
Component resolved from Trusted Renderer Catalog
5
Props validated against component contract schema
6
Component mounted in workspace pane
7
User action → deterministic backend handler (NOT the agent)
8
Result fed back as TOOL_RESULT into AG-UI thread

My Addition — Authorization Gate Architecture 

// Component Authorization Gate
async function authorizeRender(
  event: ComponentRenderEvent,
  session: AIMSSession
): Promise<AuthDecision> {
  const { componentId, props, authToken } = event;
  const { role, permissions, threadId } = session;

  // 1. Verify authToken signed by orchestrator (not the agent)
  const verified = await verifyOrchestratorSignature(authToken);
  if (!verified) return { allow: false, reason: 'invalid_auth_token' };

  // 2. Check role × component matrix
  const allowed = AUTH_MATRIX[role]?.[componentId];
  if (!allowed) return { allow: false, reason: 'role_not_permitted' };

  // 3. Validate props schema
  const valid = validateProps(componentId, props);
  if (!valid) return { allow: false, reason: 'invalid_props' };

  // 4. Log to evidence chain
  await logToAuditChain({ threadId, componentId, role, timestamp: now() });

  return { allow: true, resolvedComponent: CATALOG[componentId] };
}

Thread Persistence

Hot/cold thread storage architecture and hydration strategy

Thread persistence replaces sessionStorage with a durable, tiered store. Redis holds active threads (last 24h); PostgreSQL archives cold threads. Inspector/Gemma threads use local SQLite and merge on sync. Hydration replays the last N state snapshots to rebuild workspace context on reconnect.

Redis (Hot)

  • • Active threads from last 24h
  • • Sub-millisecond read latency
  • • State snapshots + message heads
  • • TTL: 24h, then archive to Postgres
  • • Pub/Sub for real-time workspace sync

PostgreSQL (Cold)

  • • Full thread archive with all runs
  • • JSONB columns for flexible state
  • • Indexed by threadId, role, assetId
  • • Regulatory retention: 7 years minimum
  • • Partitioned by month for query perf

SQLite (Local)

  • • Inspector PWA only
  • • Offline-first, full R/W capability
  • • Vector clock for CRDT merge
  • • Evidence hash chain local ledger
  • • Sync delta on reconnect

Inspector · Gemma-4-2B-IT Intent Catalog

Copilot Addition · New Layer

On-device local agent · Offline-first · Cloud escalation capable · 10 intents

Gemma-4-2B-IT runs entirely on-device (INT4 quantized, ~2.1GB RAM). It has its own intent catalog scoped to field work. When an intent exceeds Gemma's confidence threshold or requires regulatory sign-off, it emits an EscalationIntent routed to the appropriate cloud workspace. All intents work offline; escalations queue and transmit on connectivity restore.

interface BaseIntentRequest {
  // AG-UI identifiers
  threadId: string;        // UUIDv7 — AG-UI thread

  // Session context
  intent: string;          // e.g. 'planner_session_brief'
  role: AIMSRole;          // 'planner'|'reviewer'|'supervisor'|'inspector'

  // Evidence chain
  sessionNonce: string;    // Single-use — prevents replay
  requestedAt: string;     // ISO 8601 UTC

  // Gemma-specific (Inspector only)
  gemmaContext?: {
    modelVersion: string;  // '4-2B-IT-INT4'
    inferenceMs: number;   // Local latency
    confidence?: number;   // 0–1
    offlineDuration?: number; // Seconds offline
  };

  // Intent-specific payload
  payload: Record<string, unknown>;
}

Evidence Chain Governance Contract

Copilot Addition

Every artifact — photograph, measurement, agent output, approval, override — is committed to an immutable hash chain. Chain integrity is verified at sync, at review, and at regulatory submission. A broken chain is an immediate CRITICAL alert.

EvidenceRecord Schema

interface EvidenceRecord {
  recordId: string;          // EV-{YYYYMMDD}-{seq}
  workOrderId: string;
  assetId: string;

  // Capture
  capturedAt: string;        // ISO 8601
  capturedBy: string;        // userId or 'GemmaAgent'
  deviceId: string;
  gps: { lat: number; lng: number; accuracyM: number };

  // Artifact
  artifactType: 'image' | 'measurement' | 'agent_output' | 'approval' | 'override';
  artifactPayload: Record<string, unknown>;

  // Hash chain
  contentHash: string;       // SHA-256(artifactPayload)
  prevHash: string;          // Hash of previous record in chain
  chainHash: string;         // SHA-256(contentHash + prevHash)

  // Classification (if Gemma)
  gemmaClassification?: {
    defectType: string;
    severity: 'nominal'|'low'|'moderate'|'high'|'critical';
    confidence: number;
    modelVersion: string;
    inferenceMs: number;
  };

  // Approval state
  status: 'captured'|'pending_review'|'approved'|'flagged'|'rejected';
  approvedBy?: string;
  approvedAt?: string;
  approvalHash?: string;     // Hash of approval action

  // Sync
  syncStatus: 'local'|'synced'|'conflict';
  syncedAt?: string;
}

Offline-First CRDT Sync Contract

Copilot Addition

The Inspector PWA is offline-first. All writes go to local SQLite first. Sync uses a vector clock CRDT — each device has a vector of {deviceId: lamportClock} pairs. On reconnect, a delta sync transmits only records created or modified since the last successful sync.

SyncDelta Message

interface SyncDelta {
  deviceId: string;
  syncSessionId: string;

  // CRDT vector clock
  deviceVector: Record<string, number>;    // {deviceId: lamportClock}
  serverVector: Record<string, number>;    // Server sends back its vector

  // Records
  records: EvidenceRecord[];               // New/modified since last sync
  escalations: GemmaEscalationEnvelope[];  // Queued escalations
  workOrderAcks: WorkOrderAck[];           // Status updates for WOs

  // Conflict hints
  conflicts?: SyncConflict[];

  // Priority ordering
  priorityOrder: string[];  // escalationIds + recordIds, CRITICAL first
}

interface SyncConflict {
  recordId: string;
  conflictType: 'concurrent_edit' | 'approval_conflict' | 'hash_mismatch';
  localVersion: EvidenceRecord;
  serverVersion: EvidenceRecord;
  resolution: 'server_wins' | 'human_required' | 'merge';
}

Agent-to-Agent (A2A) Handoff Envelope

Copilot Addition

When Gemma (edge) hands off to a cloud agent, or when cloud agents chain (e.g., AnalysisAgent → ReviewAgent), a standardized A2A envelope carries context, prevents duplication, and ensures the receiving agent has everything it needs.

interface A2AHandoff {
  handoffId: string;         // UUIDv7
  fromAgent: string;         // 'GemmaAgent-{deviceId}' | 'AnalysisAgent'
  toAgent: string;           // 'ReviewAgent' | 'SupervisorAgent'
  handoffReason: string;     // Human-readable

  // Thread continuity
  originThreadId: string;    // Inspector's local thread
  targetThreadId?: string;   // Cloud thread (assigned on delivery)

  // Context package
  context: {
    workOrderId: string;
    assetId: string;
    findingSummary: string;
    evidenceRecordIds: string[];
    regulatoryRefs: string[];
    fromAgentReasoning: string;    // Why this handoff?
    fromAgentConfidence: number;
    suggestedActions: string[];    // Agent's suggestions (not authoritative)
  };

  // Governance
  handoffHash: string;       // SHA-256 of context
  requiresHumanAction: boolean;
  requiresHumanActionReason?: string;
  sentAt: string;
  ttl: number;               // Handoff expires after N seconds
}

Architecture Additions — Copilot Suggestions

8 additions

Additions to the VS Code Copilot proposed sequence

#1→ Sequence Step 7

Inspector / Gemma Offline Intent Catalog

The field edge is a first-class citizen in the architecture. Gemma-4-2B-IT needs its own intent catalog, component registry, and A2A envelope. Without this, the offline-to-online escalation path has no typed contract.

#2→ Sequence Step 8

CRDT Offline Sync Contract

sessionStorage as truth is already flagged for replacement — but the Inspector PWA needs more than that. A CRDT vector clock sync model handles offline periods of 4–24h without data loss or conflict ambiguity.

#3→ Sequence Step 10

Evidence Chain Governance Contract

Hash-chaining is referenced in the UI but not yet defined as an architectural contract. Defining EvidenceRecord, contentHash, chainHash, and tamper hooks early prevents expensive retrofits when regulatory auditors arrive.

#4→ Sequence Step 7

GemmaEscalationEnvelope

Escalations from Gemma carry more context than cloud-to-cloud escalations: offline duration, local confidence, regulatory reference, and model version. Standardize this envelope before connecting the first Inspector intent.

#5→ Sequence Step 3 (add gate now)

Component Authorization Matrix

AG-UI lets agents propose UI. The authorization gate must be server-enforced, not client-enforced. Define the matrix before wiring production intents — retrofitting authorization is costly and risky.

#6→ Sequence Step 9

Reviewer + Supervisor Full Intent Catalogs

The VS Code Copilot sequence focuses on Planner. Reviewer and Supervisor intents share the same transport and should be catalogued in parallel to avoid divergent patterns. Especially: approve_evidence_record (SIGN trust) and supervisor_override (OVERRIDE trust) need careful design.

#7→ Sequence Step 14

Multi-modal Intent Router

The Inspector app takes text, voice, and image inputs. A single intent router that normalizes these to the same BaseIntentRequest envelope prevents N separate handlers. Voice: transcribe → intent. Image: classify → intent. Text: NLU → intent.

#8→ Sequence Step 15

Regulatory Schema Local Cache

Gemma's regulatory lookup intent (API 653, PSSR, ATEX) requires a local document cache on the Inspector device. Define the schema cache format (e.g., SQLite FTS table), update cadence (sync on connect), and Gemma RAG retrieval contract before the Inspector goes to production.
