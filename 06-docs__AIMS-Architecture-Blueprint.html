Executive Summary

AIMS (Asset Integrity Management System) adopts a three-layer agentic architecture that enforces a strict separation between agent intent, workspace rendering, and deterministic execution. The three primary layers are:

  1. AG-UI — Persistent agent session and event transport (threadId, SSE events, message snapshots, state)
  2. A2UI — Dynamically generated workspaces proposed by agents, rendered from a trusted component catalog
  3. AIMS Trusted Renderer Catalog — The deterministic, role-authorized component registry that agents may reference but never bypass
Core Principle (VS Code Copilot)
The agent may propose the UI and action. Authorization and execution remain in deterministic backend services.

An additional fourth sub-layer is introduced via the Inspector layer: Gemma-4-2B-IT, an on-device language model that operates fully offline and escalates to cloud orchestrators via a governed A2A envelope when confidence thresholds are breached or cloud intervention is required. This offline-capable intelligence layer enables field inspection in zero-connectivity environments while preserving full evidence integrity.

This document is the authoritative engineering reference synthesizing recommendations from VS Code Copilot and supplementary Copilot additions into a single, actionable blueprint. It defines all intent catalogs, typed contracts, the component registry, the authorization matrix, and the full 15-step implementation roadmap.

Section 4: AG-UI Transport Layer

4.1 Core Events

All events are transmitted via Server-Sent Events (SSE) over the AG-UI protocol. The event stream is keyed by threadId and multiplexed across concurrent runs within the same thread. Events are immutable once emitted and are persisted to the audit trail.

Event Direction Key Payload Fields Trigger
RUN_STARTED Server→Client threadId, runId, agentId, role, timestamp Intent dispatched
TEXT_DELTA Server→Client runId, delta, chunkIndex Agent streaming
TOOL_CALL Server→Client runId, toolName, args, callId Agent action
TOOL_RESULT Client→Server callId, result, status Action complete
STATE_SNAPSHOT Server→Client threadId, state, version State hydration
RUN_FINISHED Server→Client runId, finalState, duration Run complete
COMPONENT_RENDER Server→Client componentId, props, authToken A2UI render signal
ESCALATION_QUEUED Server→Client escalationId, priority, routing Threshold breach
SYNC_DELTA Bidirectional deviceId, records[], crdt_vector Offline sync merge

4.2 Thread Persistence Strategy

sessionStorage is replaced by a two-tier durable thread store. This ensures that no agent context is lost across browser refreshes, device reconnects, or application restarts.

  • Hot tier — Redis: Threads active within last 24 hours; sub-millisecond lookup by threadId
  • Cold tier — PostgreSQL: Archived threads; full audit trail; queryable by threadId, userId, agentId, and date range
  • Inspector local — SQLite: On-device persistence for Gemma threads; merged to cloud on sync via vector clock reconciliation
  • Hydration on reconnect: Replay last N STATE_SNAPSHOT events to restore full workspace context without re-executing agent logic

Section 5: A2UI Workspace Layer

5.1 Render Pipeline

The A2UI render pipeline enforces a strict authorization gate between agent intent and component rendering. No component may be mounted without a valid orchestrator-signed token and role-based approval. The eight-step pipeline is:

  1. Agent resolves intent → emits componentId
  2. COMPONENT_RENDER event emitted via AG-UI SSE
  3. Authorization Gate checks: role × componentId × action → ALLOW / DENY
  4. Component resolved from Trusted Renderer Catalog
  5. Props validated against component contract schema (Zod)
  6. Component mounted in workspace pane with fade transition
  7. User action → deterministic backend handler (never the agent directly)
  8. Result fed back as TOOL_RESULT into AG-UI thread
Security Invariant
At no point does an agent directly execute an action on behalf of the user. All user-visible actions flow through the Authorization Gate and deterministic backend handlers. The agent may propose; it may never execute.

5.2 Authorization Gate

async function authorizeRender(
  event: ComponentRenderEvent,
  session: AIMSSession
): Promise<AuthDecision> {
  const { componentId, props, authToken } = event;
  const { role, threadId } = session;

  // 1. Verify authToken signed by orchestrator (not the agent)
  const verified = await verifyOrchestratorSignature(authToken);
  if (!verified) return { allow: false, reason: 'invalid_auth_token' };

  // 2. Check role × component authorization matrix
  const allowed = AUTH_MATRIX[role]?.[componentId];
  if (!allowed) return { allow: false, reason: 'role_not_permitted' };

  // 3. Validate props schema (Zod)
  const valid = validateProps(componentId, props);
  if (!valid) return { allow: false, reason: 'invalid_props_schema' };

  // 4. Log render event to evidence chain audit log
  await logToAuditChain({ threadId, componentId, role, timestamp: now() });

  return { allow: true, resolvedComponent: CATALOG[componentId] };
}

All intents across all roles share a common typed envelope. This ensures consistent authentication, audit trail generation, and Gemma context propagation. Intent-specific payloads are contained in the payload field.

Section 14: Agent-to-Agent (A2A) Handoff Envelope

The A2A handoff envelope governs all structured transfers of context between agents, including Gemma-to-cloud and cloud-to-cloud handoffs. The envelope preserves thread continuity across agent boundaries and includes a governance hash to prevent tampering in transit.

interface A2AHandoff {
  handoffId: string;              // UUIDv7
  fromAgent: string;              // 'GemmaAgent-{deviceId}' | 'AnalysisAgent'
  toAgent: string;                // 'ReviewAgent' | 'SupervisorAgent'
  handoffReason: string;

  // Thread continuity
  originThreadId: string;
  targetThreadId?: string;        // Assigned by cloud on delivery

  // Context package
  context: {
    workOrderId: string;
    assetId: string;
    findingSummary: string;
    evidenceRecordIds: string[];
    regulatoryRefs: string[];
    fromAgentReasoning: string;
    fromAgentConfidence: number;
    suggestedActions: string[];   // Suggestions only — not authoritative
  };

  // Governance
  handoffHash: string;            // SHA-256(context)
  requiresHumanAction: boolean;
  requiresHumanActionReason?: string;
  sentAt: string;
  ttl: number;                    // Seconds before handoff expires
}
Thread Continuity Note
The originThreadId from the Inspector/Gemma thread is preserved throughout the handoff chain. The receiving cloud agent creates a new targetThreadId that references the origin, maintaining a complete, queryable ancestry trail from field capture to regulatory submission.

Section 16: Authorization Matrix

Authorization Notes

  • Inspector ACTION denied: Inspector proposes via Gemma inference; execution requires cloud agent confirmation after sync. This is not a user-experience limitation — it is a governance invariant.
  • Inspector SIGN denied: Field co-sign is available only after Reviewer approval of the full evidence record. Inspector acknowledgement at capture does not constitute approval.
  • Planner ESCALATE conditional: Scheduling conflicts only. Planners cannot escalate evidence decisions or regulatory findings — those are Reviewer/Supervisor jurisdiction.
  • All matrix entries are server-enforced: The Authorization Gate validates role × component × action on every COMPONENT_RENDER event. Client-side enforcement is advisory only.

A single entry point handles all input modalities — text input, voice command, and image capture — and routes to the correct agent and intent based on content type and role context. This eliminates per-modality code paths and ensures consistent authorization regardless of how the user initiates an action.

18.2 Cache Strategy

  • Schemas cached as structured JSON on Inspector device at last sync
  • Version hash checked on each sync — schema updated if server version differs
  • Gemma uses RAG (retrieval-augmented generation) over cached schemas for regulatory_lookup intent
  • Regulatory references surfaced in Gemma output include section identifiers (e.g. API 653 §4.4.1) and are included in the GemmaEscalationEnvelope
  • Cloud Reviewer/Supervisor always queries the authoritative version — local cache is advisory only for field use
  • Schema updates are propagated to all Inspector devices within the next sync cycle following a cloud-side schema version bump

Appendix A: Glossary

TermDefinition
AG-UI Agent-UI protocol — persistent session, SSE event transport, thread model for agent–workspace communication
A2UI Agent-to-UI — dynamic workspace rendering from the trusted component catalog; agents propose, the gate authorizes
A2A Agent-to-Agent — structured handoff envelope for context transfer between agents across trust boundaries
CRDT Conflict-free Replicated Data Type — enables offline-first sync without central coordination using vector clocks
threadId UUIDv7 identifier for an AG-UI conversation thread — time-ordered, persisted in Redis/PostgreSQL
runId Identifier for a single agent run within a thread; scopes all events for that run
GemmaEscalationEnvelope Typed contract governing Inspector→Cloud escalations including full local inference context and routing metadata
Trusted Renderer Catalog The finite, server-enforced set of A2UI components agents may reference — enforced by the Authorization Gate
Vector Clock CRDT mechanism tracking per-device logical time (Lamport clock) for conflict-free merge on sync
Chain Hash SHA-256(contentHash + prevHash) — links each evidence record to its predecessor for tamper detection
Gemma-4-2B-IT On-device language model running INT4 quantized (~2.1 GB RAM); handles all Inspector intents fully offline
sessionNonce Single-use token included in every BaseIntentRequest to prevent replay attacks
LWW Last-Write-Wins — conflict resolution strategy using Lamport clock for field-level CRDT merge
RAG Retrieval-Augmented Generation — Gemma's mechanism for querying cached regulatory schemas during offline regulatory_lookup intents
